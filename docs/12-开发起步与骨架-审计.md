## C. 必改项

### C1. 工程结构 — 缺少 .sln 解决方案文件

**问题**：任务板 S0-01 要求创建 SeatBook.sln、S0-03 要求添加项目引用到解决方案，但仓库中不存在任何 `.sln` 文件。`dotnet build` 虽然能通过（直接编译 .csproj），但 VS 中无法直接打开解决方案，对后续多项目扩展（如添加测试项目）不友好。

**修复要求**：在 `SeatBook/` 上级目录（即 `D:\AiWeb\3`）创建 `SeatBook.sln` 并将 `SeatBook\SeatBook.csproj` 加入解决方案。
```bash
dotnet new sln -n SeatBook -o .
dotnet sln add SeatBook\SeatBook.csproj
```

### C1b. README §60-77 "待生成"目录结构与实际不符

**问题**：README §62-76 描述的"后续计划 / 待生成"结构为 `SeatBook.sln` + `SeatBook/SeatBook/` 嵌套路径，但实际项目为 `SeatBook/` 扁平结构（Controllers/、Models/、Services/、Data/、Views/、wwwroot/ 直接在 SeatBook 下）。实习生读 README 会认为这些目录尚未创建（标注为"待生成"），实际已存在。

### C2. Program.cs Middleware 顺序 — UseAuthorization 应在 UseSession 之后

**问题**：`Program.cs:33` 的 `app.UseAuthorization()` 位于 `app.UseSession()`（第 34 行）之前。ASP.NET Core 中间件管道按注册顺序执行，Session 数据在授权判断时不可用（虽然当前项目未使用基于 Session 的授权策略，但架构上不正确，后续如果增加基于 Session 的策略会引入隐式 Bug）。

**修复要求**：交换两行顺序：
```cs
app.UseSession();
app.UseAuthorization();
```

### C3. 缺少 EF Core 首次迁移

**问题**：阶段 12 的放行标准和检查重点均要求"已创建首次迁移，并完成首次建库建表"。当前仅使用了 `EnsureCreated()`（Program.cs:44），无 `Migrations/` 目录。`EnsureCreated()` 无法增量管理数据库版本，且每次启动都会检查架构（轻度开销）。E FCore 迁移方案应作为骨架的一部分。

**修复要求**：生成首次迁移（可在 Sprint 1 初期完成，不阻塞当前阶段放行）：
```bash
dotnet ef migrations add InitialCreate
```
并在 Program.cs 中将 `EnsureCreated()` 替换为：
```cs
db.Database.Migrate();
```

**不阻塞 Sprint 1 的说明**：当前 `EnsureCreated()` + `Seed()` 方案功能完整，首次迁移可在 Sprint 1 启动时补做，不影响用户端编码。
