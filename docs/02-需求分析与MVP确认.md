# 图书馆座位预约系统 · 需求分析与 MVP 确认

## 1. 需求来源与分析方法

本需求分析基于《01-项目立项单》展开。分析方法如下：

1. **用户故事映射** — 将立项单中的功能列表转化为用户故事，识别核心痛点。
2. **MoSCoW 优先级矩阵** — 对每个功能按 Must / Should / Could / Won't 分级。
3. **MVP 切割** — 以"一用户能走通预约完整链路 + 管理员能做基础维护"为底线，确定本期 MVP 范围。
4. **场景推演** — 模拟学生和管理员的典型使用场景，验证功能是否完整。

## 2. 用户角色与痛点分析

| 角色 | 核心痛点 | 对应目标 |
|------|----------|----------|
| 学生 | 到馆发现无空座，或座位被占但不使用 | 提前查座、预约、管理预约 |
| 管理员 | 无法掌握座位使用情况，座位维护靠人工 | 管理座位、查看预约数据 |

## 4. MVP 范围确认

### 4.1 MVP 包含（P0 全部 + P1 中必要的辅助功能）

**用户端（5 页）：**
- 首页（推荐座位、快速入口）
- 座位列表（区域/楼层筛选、状态展示）
- 座位详情（信息展示 + 预约入口）
- 预约提交（日期 + 时段选择、确认提交）
- 我的预约（预约列表 + 取消操作）

**管理端（4 页）：**
- 管理员登录
- 座位管理（CRUD）
- 预约管理（查看所有预约、状态筛选、手动处理取消）
- 统计页（预约总数、每日预约量 — 可用表格而非图表）

**通用（3 项）：**
- 模拟登录切换（无需密码体系）
- 数据库 Seed Data
- Bootstrap 统一布局

### 4.2 MVP 不包含（已明确排除）

排除项同 P3 — Won't Have，共 7 项（US17 ~ US23）。

## 5. 删减项 / 暂缓项（相对立项单）

以下功能在立项单中出现，但经需求分析后建议暂缓或降级：

| 功能 | 原状态 | 调整后 | 理由 |
|------|--------|--------|------|
| 统计图表（柱状图） | 本期完成 | 暂缓为 P2 | 可以用表格替代，图表不是闭环必需 |
| 预约管理-手动处理 | 本期完成 | 降级为 P1 | MVP 可先只做查看+筛选，手动处理优先级较低 |
| 时段热度统计 | 本期统计页包含 | 降级为 P2 | MVP 统计页先只展示预约总数和每日预约量 |

## 6. 需求评审结论

| 评审项 | 结论 | 说明 |
|--------|------|------|
| 需求完整性 | 通过 | 覆盖学生完整预约链路 + 管理员基础维护 |
| MVP 边界 | 清晰 | 以 P0 为界，不含支付、App、硬件联动等超纲内容 |
| 可行性 | 通过 | 10 页以内，ASP.NET Core MVC 单人 4 周可行 |
| 优先级合理性 | 通过 | P0 全是核心链路，无遗漏无冗余 |
| 暂缓项合理性 | 通过 | 暂缓项不影响 MVP 业务闭环 |

**评审决定：需求分析与 MVP 范围确认通过，可以进入 PRD Lite 阶段。**
