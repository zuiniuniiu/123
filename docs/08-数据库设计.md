# 图书馆座位预约系统 · 数据库设计

## 1. 数据设计目标

- 用最少的表支撑 9 页功能和两条核心链路
- 不建多余的表，不设冗余字段
- 查询路径清晰，单表或两表 JOIN 能覆盖全部需求
- 唯一约束在数据库层面保证"同一座位同一日期同一时段"不重复预约
- Seed Data 启动即用，演示数据覆盖全部状态
- 本期不做 Migration，用 `EnsureCreated()` 建库

## 2. 核心实体清单

以前序文档（PRD-Lite + 系统设计说明）为准，共 **4 个实体 / 4 张表**：

| 序号 | 实体 | 对应表 | 说明 |
|------|------|--------|------|
| 1 | SimulatedUser | SimulatedUsers | 体验账号，用于用户端模拟登录下拉框 |
| 2 | Seat | Seats | 座位信息，含区域、楼层、状态 |
| 3 | Reservation | Reservations | 预约记录，关联座位和用户 |
| 4 | Admin | Admins | 管理员账号，用于后台登录 |

**不使用额外实体的原因：**
- 不建 `SeatArea` 表：Area 仅为座位的一个分类属性（3~5 个固定值），无需独立建表，直接存 `Seat.Area` 字符串即可。筛选时用 `DISTINCT Area` 查询。
- 不建 `TimeSlot` 表：时段固定为 3 个（08:00-12:00 / 14:00-18:00 / 18:00-21:00），用字符串存 `Reservation.TimeSlot` 即可，无需独立建表也不做枚举表。
- 不建 `ReservationLog` 表：本期不需要操作日志跟踪，预约状态变化以当前 `Reservation.Status` 为准。
- 不建 `User` 表：体验账号体系简单，`SimulatedUser` 只做展示用，预约记录中存用户名 `UserName` 字符串即可。

## 3. 关系说明与 ER 图

### 实体关系

```
SimulatedUser (1) ──── (N) Reservation (N) ──── (1) Seat

Admin（独立表，与其他实体无外键关联）
```

### 关系详情

| 关系 | 左端 | 右端 | 外键 | 约束 |
|------|------|------|------|------|
| 一对多 | SimulatedUser | Reservation | Reservation.UserName ↔ SimulatedUser.Name | 逻辑关联，不设物理外键 |
| 一对多 | Seat | Reservation | Reservation.SeatId → Seat.Id | 物理外键，CASCADE 关闭（不允许级联删除） |

**外键说明：**
- `Reservation.SeatId` 设置物理外键 → `Seat.Id`，删除座位时必须先检查无关联预约
- `Reservation.UserName` 仅做逻辑关联（存字符串），不设外键。原因是 SimulatedUser 表仅为展示用，Name 字段不是主键，且 UserName 只存文本不参与复杂关联查询

CREATE TABLE Reservations (
    Id INT IDENTITY(1,1) PRIMARY KEY,
    SeatId INT NOT NULL,
    UserName NVARCHAR(50) NOT NULL,
    ReservationDate DATE NOT NULL,
    TimeSlot NVARCHAR(20) NOT NULL,

CREATE INDEX IX_Reservations_Status ON Reservations(Status);
CREATE INDEX IX_Reservations_Date ON Reservations(ReservationDate);

## 5. 关键字段说明

| 字段 | 设计理由 | 注意事项 |
|------|----------|----------|
| `Seat.Status` 用 int 枚举 | 状态只有 3 个值，int 比 enum 字符串更小更快 | 前端显示时映射为中文标签；不删除枚举值，只能追加 |
| `Reservation.Status` 用 int 枚举 | 同上，3 个状态值 | 0=已预约是核心活跃状态，1=已完成是终态，2=已取消也是终态 |
| `Reservation.TimeSlot` 用 string | 时段固定 3 个，字符串可读性好，无需查字典表 | 前端展示和存储用相同格式，不做转换 |
| `Reservation.ReservationDate` 用 DATE | 仅存日期不存时间，查询按天过滤效率高 | C# 中用 `DateOnly` 或 `DateTime.Date` |
| `Reservation.CreatedAt` 用 DATETIME2 | 记录创建时间戳，用于"我的预约"倒序排列 | 默认值 `GETDATE()` |
| `Seat.Area` 和 `Seat.Floor` 为独立字段 | 虽然是同质数据（区域包含楼层信息），但在 MVP 中分开更清晰，筛选时可独立使用 | 如果后期区域变多，可拆为 Area 表，本期不做 |
| `Seat.Description` 允许空 | 座位描述为可选信息 | MVC 表单中标注为选填 |
| `Reservation.UserName` 存字符串 | 不设外键，SimulatedUser 仅为展示参考 | 删除体验账号不影响已有预约记录 |

## 6. 状态字段设计

### 6.1 Seat.Status 枚举

| 值 | 名称 | 显示文案 | 含义 |
|----|------|----------|------|
| 0 | Available | 空闲 | 可被预约 |
| 1 | Booked | 已预约 | 已被某个用户预约 |
| 2 | Maintenance | 维护中 | 管理员标记为不可用 |

### 6.2 Reservation.Status 枚举

| 值 | 名称 | 显示文案 | 含义 |
|----|------|----------|------|
| 0 | Active | 已预约 | 预约生效中 |
| 1 | Completed | 已完成 | 管理员手动标记为已使用 |
| 2 | Cancelled | 已取消 | 用户或管理员取消 |

### 6.3 状态联动规则（数据层面）

| 触发操作 | Reservation 变化 | Seat 变化 |
|----------|-----------------|-----------|
| 用户提交预约 | 插入 Status=0 | Status 置为 1（已预约） |
| 用户取消预约 | 对应记录 Status 置为 2（已取消） | Status 置为 0（空闲） |
| 管理员标记完成 | 对应记录 Status 置为 1（已完成） | Status 不变化（保持 1） |
| 管理员切换座位为"维护中" | 不变化 | Status 置为 2（维护中） |
| 管理员切换座位为"空闲" | 不变化 | Status 置为 0（空闲） |

**关键规则：** 以上所有状态联动必须在同一个 `DbContext.SaveChangesAsync()` 的事务中完成，保证数据一致性。

## 8. 统计字段与统计口径

### 8.1 统计口径定义

| 统计项 | 口径 | SQL 示例 |
|--------|------|----------|
| 座位总数 | Seats 表全部记录数 | `SELECT COUNT(*) FROM Seats` |
| 今日预约数 | Reservations 中 ReservationDate=今天 且 Status=0 的记录数 | `SELECT COUNT(*) FROM Reservations WHERE ReservationDate=CAST(GETDATE() AS DATE) AND Status=0` |
| 本月预约数 | Reservations 中 ReservationDate 属于当月 且 Status=0 的记录数 | `SELECT COUNT(*) FROM Reservations WHERE ReservationDate BETWEEN @monthStart AND @monthEnd AND Status=0` |
| 各时段分布 | 按 TimeSlot 分组统计全部 Reservations 的预约次数 | `SELECT TimeSlot, COUNT(*) FROM Reservations GROUP BY TimeSlot ORDER BY COUNT(*) DESC` |

### 8.2 统计实现方式

所有统计值实时查询，不做缓存、不做汇总表。原因是数据量极小（MVP 阶段 < 100 条预约），实时查询即可。

```csharp
// StatisticsService.GetStatisticsAsync() 伪代码
var totalSeats = await _context.Seats.CountAsync();
var today = DateTime.Today;
var todayReservations = await _context.Reservations
    .CountAsync(r => r.ReservationDate == today && r.Status == 0);
var monthStart = new DateTime(today.Year, today.Month, 1);
var monthEnd = monthStart.AddMonths(1).AddDays(-1);
var monthReservations = await _context.Reservations
    .CountAsync(r => r.ReservationDate >= monthStart && r.ReservationDate <= monthEnd && r.Status == 0);
var timeSlotDistribution = await _context.Reservations
    .GroupBy(r => r.TimeSlot)
    .Select(g => new { TimeSlot = g.Key, Count = g.Count() })
    .ToListAsync();
```

## 9. 数据一致性与约束说明

### 9.1 数据库层级约束

| 约束类型 | 具体约束 | 作用 |
|----------|----------|------|
| 主键约束 | 4 张表均有 Id INT IDENTITY PRIMARY KEY | 行唯一标识 |
| 外键约束 | Reservation.SeatId → Seats.Id | 保证预约记录指向存在的座位 |
| 唯一过滤索引 | Reservations(SeatId, ReservationDate, TimeSlot) WHERE Status=0 | 防止同一座位同一日期同一时段被重复预约（核心业务规则） |
| 非空约束 | 所有业务字段均 NOT NULL（除 Description） | 保证数据完整性 |
| 默认值 | Reservation.Status=0, Reservation.CreatedAt=GETDATE(), Seat.Status=0 | 减少代码中遗漏赋值的可能 |

### 9.2 应用层约束（Service 中实现）

| 约束 | 实现位置 | 说明 |
|------|----------|------|
| 预约时 Seats.Status 必须为 0 | ReservationService.CreateAsync 开头 | 先查询 Seat.Status，非空闲则返回失败 |
| 取消时 Reservation.Status 必须为 0 | ReservationService.CancelAsync | 只允许取消"已预约"状态的记录 |
| 删除座位前检查无关联预约 | AdminService.DeleteSeatAsync | 查 Reservations 表是否有该 SeatId 的记录 |
| 标记完成时 Reservation.Status 必须为 0 | ReservationService.MarkCompletedAsync | 只允许标记"已预约"状态的记录 |

### 9.3 数据一致性保障

```
预约操作（ReservationService.CreateAsync）事务流程：

1. BEGIN TRANSACTION
2. SELECT Seat.Status FROM Seats WHERE Id=@seatId  -- 检查座位状态
3. IF Seat.Status != 0 → ROLLBACK, return false
4. 检查唯一索引冲突（EF Core 插入时会自动检查，或用 AnyAsync 预检）
5. INSERT INTO Reservations (SeatId, UserName, ReservationDate, TimeSlot, Status=0, CreatedAt=now)
6. UPDATE Seats SET Status=1 WHERE Id=@seatId
7. COMMIT TRANSACTION
```

### 9.4 级联删除策略

| 主表 | 从表 | 级联策略 | 说明 |
|------|------|----------|------|
| Seats | Reservations | NO ACTION（禁止级联删除） | 删除座位前必须在代码层检查无关联预约 |
| SimulatedUser | Reservation | 无外键，不涉及 | — |

## 11. 默认补足项 / 当前假设

本期数据库设计以 `docs/03-PRD-Lite.md` 第 11 章的 4 个核心数据对象为准。以下为设计过程中做的补充决策和假设：

### 11.1 补充决策

| 决策项 | 选择 | 依据 |
|--------|------|------|
| `Reservation.UserName` 不设物理外键 | 存字符串，不关联 SimulatedUsers.Id | SimulatedUsers 仅用于展示，删除体验账号不应影响已有预约 |
| `Reservation.ReservationDate` 用 DATE 而非 DATETIME | 预约只需要日期，精确到天即可 | 查询按日期过滤更简单高效 |
| 唯一约束用过滤索引而非 UNIQUE 约束 | `WHERE Status=0` 的过滤索引只约束活跃预约 | 允许同一座位同一时间存在已取消和已完成的旧记录 |
| 不创建 Area 独立表 | Area 作为 Seat 的字符串字段 | 仅 3~5 个固定区域值，独立表增加 JOIN 开销 |
| 不创建 TimeSlot 独立表 | TimeSlot 作为 Reservation 的字符串字段 | 仅 3 个固定时段值，不需要字典表 |
| 不创建 ReservationLog 表 | 不做预约操作日志 | MVP 不需要，管理员可通过预约状态变化追溯 |

### 11.2 当前假设

| 假设 | 如果假设不成立时的调整 |
|------|----------------------|
| 座位总数 < 100 个 | 超过时需在座位列表和管理端加分页 |
| 预约记录 < 1000 条 | 超过时需在管理端加日期范围筛选和分页 |
| 时段固定为 3 个，不新增 | 新增时段只需在代码中加选项，数据库不需改动 |
| 区域固定为 3~5 个，不新增 | 新增区域直接用 Seed Data 插入，Area 可扩展 |
| 同时只有一个管理员 | 多管理员时需扩展 Admin 表字段（角色、状态、最后登录时间） |
| 体验账号个数 < 10 个 | 超过时下拉框需改为搜索模式 |
| 不出现高并发预约 | 如果出现并发，需在事务中加 `UPDLOCK` 行锁提示 |
