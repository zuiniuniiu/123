## B. 发现的问题清单

| # | 严重度 | 问题描述 | 涉及位置 |
|---|--------|----------|----------|
| 1 | 🟡 建议 | **AdminController 依赖 3 个 Service，职责边界偏宽**：`AdminController` 注入了 `IAdminService` + `ISeatService` + `IReservationService`，管理了 3 个 Service。虽然对于 4 页管理端可接受，但 Seat 和 Reservation 的操作本应通过各自的 Controller 完成 | §6.4 |
| 2 | 🟡 建议 | **Session["IsAdmin"] 类型表述不一致**：§5.3 的 Session 方案表中写 `IsAdmin | true(bool)`，但下方代码用的是 `SetString("IsAdmin", "true")` — ASP.NET Core Session 的 `SetString` 存的是 string，不是 bool。容易引起后续编码误解 | §5.3 |
| 3 | 🟡 建议 | **Service 接口与实现分离对 9 页项目是否过度**：`Services/Interfaces/` + `Services/Implementations/` 两层目录。对于单人课堂项目且无单元测试需求（无 Mock），可以直接使用具体类并依赖注入。可简化但不算违规 | §3.1 |
| 4 | 🟢 提示 | **未定义 `[AdminAuth]` ActionFilter 的具体实现**：§6.4 用 `[AdminAuth]` 标注了拦截需求，但文档中未定义该 Filter 的代码位置或实现逻辑（虽然 §5.3 有 Session 检查逻辑的描述） | §6.4 |
| 5 | 🟢 提示 | **取消预约的 POST 路由未定义**：§6.3 `POST Cancel(int id)` 没有定义对应路由（是 `/Reservation/Cancel/{id}` 还是带 `[HttpPost]` 的表单提交？），需要在编码时明确 | §6.3 |
| 6 | 🟢 提示 | **未涉及错误处理策略**：文档未说明 404（座位不存在）、500、模型验证失败等场景的全局处理方式。对于课堂项目可以在编码阶段补充 | 全文 |
| 7 | 🟢 提示 | **统计 Service 返回 `StatisticsData` 未定义**：§6.5 的 `IAdminService.GetStatisticsAsync()` 返回了 `StatisticsData`，但文档中未给出这个类的字段定义，编码时需要补 | §6.5 |

## D. 建议优化项（3 项）

1. **统一 Session["IsAdmin"] 的表述**：将 §5.3 方案表中的 `true(bool)` 改为 `"true"(string)`，与下方代码一致。
2. **为 AdminController 减负（可选）**：可在 `AdminController` 中只保留 `IAdminService`，将座位管理相关 Action 委托给 `SeatController`（允许 POST），将预约管理委托给 `ReservationController`（允许管理员操作），让 AdminController 只做登录和统计。
3. **补全 `[AdminAuth]` 的定义**：在 §9 之后补充一段"附录：AdminAuth ActionFilter"，给出 Filter 的完整代码（20 行以内），避免编码时临时查 Session 写法。
