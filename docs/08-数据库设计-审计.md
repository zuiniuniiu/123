## C. 必改项（1 项）

### C1. 明确 ReservationDate 的 C# 类型与 EF 映射

**问题**：表定义写 SQL 类型为 `date`，但 03-PRD 写 `DateTime`。.NET 8 同时支持 `DateTime` 和 `DateOnly`，需明确选择并给出 EF Core 配置。

**修改建议**：在 §5.2 实体配置中补充：

```
// 方案 A：使用 DateTime（推荐，兼容性最好）
Property(r => r.ReservationDate).HasColumnType("date");

// 或方案 B：使用 DateOnly（.NET 8，语义更准确）
Property(r => r.ReservationDate).HasConversion<DateOnlyConverter>();
```

推荐方案 A（DateTime + .HasColumnType("date")），因为 EF Core 对 DateTime 支持最成熟，且与 03-PRD 的 `DateTime` 定义一致。

## D. 建议优化项（2 项）

1. **补充 Seat 删除前检查规则**：在 §8 冲突规则之后增加一段："删除座位前置检查：Service 层删除 Seat 前需查询 Reservation 表中是否存在该 SeatId 且 Status=Active 的记录，存在则拒绝删除。"
2. **标注 SimulatedUser 删除风险**：在 ER 图说明的脚注中补充："UserName 逻辑外键：当 SimulatedUser 被删除时，关联的 Reservation.UserName 不会自动清理（无 FK 约束）。当前设计不允许删除体验账号，此风险可接受。"
